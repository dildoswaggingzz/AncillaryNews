FCR-D (fast-disturbance reserve) and FFR (Fast Frequency Reserve) only exist in DK2 -- ticking either for a DK1 run is now rejected server-side (rather than silently earning nothing on those legs, as it used to). aFRR down-regulation capacity is available in both zones.

FFR currently clears at 0.0 (see shared/datasets.py) -- ticking it automatically switches this run to capacity_allocation="price_ranked" server-side, so FFR's zero price doesn't dilute the other markets' shares the way an even split would (shared/bess_simulator.py's module docstring §2).

Foresight only affects the cooptimized strategy (ignored by threshold runs). perfect foresight is an oracle -- it schedules against realised prices it could not actually have known in advance -- and is shown here only as a theoretical ceiling, not a number a real battery could earn. forecast foresight schedules against a simple, honest day-ahead persistence forecast and settles at actuals -- this is the achievable, deployable estimate.
